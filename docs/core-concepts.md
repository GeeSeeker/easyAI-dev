# 🧠 核心概念：理解 easyAI 的设计

> **预计阅读时间**：15 分钟 | **目标**：理解框架的架构哲学和核心机制

---

## 1. 设计哲学：框架即人体

easyAI 的底层设计源自一个核心比喻：

> **easyAI 框架 = 人体。AI = 大脑。** MCP/Skills/Workflows/Trellis = 身体部件。**知识图谱 + 合规机制 = 神经系统 + 血管系统。**

传统的 AI 编码工具只是给 AI 一堆独立的工具。easyAI 不同 — 它让 AI 拥有一个**有自我感知能力的完整身体**。

### 三大核心目标

| 目标         | 含义                                        | 比喻                     |
| ------------ | ------------------------------------------- | ------------------------ |
| **自我感知** | AI 随时知道框架拥有什么能力，不需要用户提醒 | 感知到身体各部位的存在   |
| **自主协调** | 面对任务，AI 自动识别需要调动哪些模块配合   | 跑步时手脚自动协调       |
| **精准迭代** | 增加新能力时，AI 知道跨模块需要修改哪些文件 | 知道血液要输送到哪些肌肉 |

### 如何实现？

- **知识图谱**（`.agents/graph/`）就是框架的**神经系统** — 让 AI 感知框架全貌
- **合规机制**（Rules + Evidence Gate）就是**血管系统** — 确保质量标准流向每个环节
- **Skills 编排链**是**肌肉记忆** — 面对不同场景自动调用正确的能力组合

## 2. 五层生长架构

easyAI 采用**五层树状架构**，从基础设施向上生长到用户体验。底层变化自动向上传播影响：

```text
┌─────────────────────────────────────────────────────────┐
│  Layer 4 · 用户体验层                                    │
│  需求交付闭环 ── 会话管理 ── 文档沉淀                      │
├─────────────────────────────────────────────────────────┤
│  Layer 3 · 角色系统层                                    │
│  PM 工作流 ── Worker 工作流                               │
├─────────────────────────────────────────────────────────┤
│  Layer 2 · 能力层                                        │
│  任务管理 ── 质量控制 ── 上下文管理 ── 知识管理 ── 框架治理  │
│  ── 外部 CLI 调度                                        │
├─────────────────────────────────────────────────────────┤
│  Layer 1 · 数据层                                        │
│  任务存储 ── 规范体系 ── 工作记忆 ── 配置管理               │
├─────────────────────────────────────────────────────────┤
│  Layer 0 · 基础设施层                                    │
│  MCP 通信 ── 配置加载 ── 文件持久化 ── Git 集成 ── 升级机制 │
└─────────────────────────────────────────────────────────┘
```

### 各层职责

| 层级                 | 职责                      | 关键组件                                          |
| -------------------- | ------------------------- | ------------------------------------------------- |
| **Layer 0** 基础设施 | 框架运行的底层基石        | MCP Server、config.yaml、Git、Manifest 升级       |
| **Layer 1** 数据层   | 结构化数据的组织与存储    | 任务目录 (tasks/)、规范文件 (spec/)、journal 日志 |
| **Layer 2** 能力层   | 框架核心业务能力          | 任务生命周期、TDD 流程、Token 预算、外部 CLI 审查 |
| **Layer 3** 角色系统 | PM 和 Worker 的工作流编排 | 17 个 Skills 的编排链路                           |
| **Layer 4** 用户体验 | 用户可感知的交付闭环      | 需求→验收的完整流程、跨会话记忆                   |

### 双向链接

每个图谱节点知道三个方向的关联：

- **向上（serves）**：为哪些上层特性提供能力
- **向下（depends_on）**：依赖哪些底层特性
- **横向（relates_to）**：与同层哪些特性有协作关系

这意味着修改任何一个特性时，AI 可以精准定位所有受影响的上下游模块。

## 3. 角色系统：PM vs Worker

easyAI 的核心创新是**双角色分工**。两个角色有严格的职责边界：

### PM（项目经理）

| 方面         | 描述                                   |
| ------------ | -------------------------------------- |
| **触发**     | `/actor-pm`                            |
| **职责**     | 需求沟通、任务规划、验收审批、资源调度 |
| **不做**     | 直接编写代码、调试 Bug、运行测试       |
| **Git 权限** | 可以 push 和 merge（需用户确认）       |
| **交互方式** | 通过约束集与 Worker 通信，不口头指令   |

### Worker（执行者）

| 方面         | 描述                                        |
| ------------ | ------------------------------------------- |
| **触发**     | `/actor-worker T001`（含任务 ID）           |
| **职责**     | 按约束集编码、测试、自检、提交验收          |
| **不做**     | 擅自扩大需求、变更技术方案、跳过验证        |
| **Git 权限** | 仅 commit，不 push、不 merge                |
| **执行模式** | 独立执行 / 团队模式（组长+组员） / 轻量修复 |

### 为什么要分开？

- **专注**：PM 专注决策，Worker 专注执行，各司其职
- **约束**：Worker 被约束集精确限制，减少意外行为
- **审计**：PM 独立审查 Worker 产出，形成质量闭环

## 4. 任务生命周期

每个任务都经历完整的生命周期：

```text
pending ──→ in_progress ──→ under_review ──→ completed ──→ archived
  │              │                │
  │              │                └──→ rejected ──→ in_progress（修复后重新提交）
  │              │
  └──────────────┴──→ cancelled（附带原因）
```

### 关键概念

**约束集**：PM 产出的不是 TODO 清单，而是**缩小 Worker 解空间的精确约束**。约束集的每一条都必须被执行，不可选择性忽略。

**Evidence Gate**：Worker 从 `in_progress` 转到 `under_review` 时的**强制验证门**。要求每个验证类别标记为 `_PASS` 或 `_NA`，拒绝 `_FAIL`。这是 MCP 层的硬性约束，AI 无法绕过。

**子任务 DAG**：支持子任务间的依赖声明和自动环检测，用于团队模式下组长拆分子任务。

## 5. 质量保障体系

easyAI 通过四层机制保障代码质量：

### 第一层：TDD 铁律

> 没有失败测试，不准写生产代码。

Worker 编码遵循 **Red-Green-Refactor** 循环：先写一个必定失败的测试 → 写最小实现让测试通过 → 重构优化。违反铁律的代码必须删除重来。

**豁免场景**（须预先声明）：纯配置/Markdown、自动生成代码、PM 标注 `[TDD-EXEMPT]` 的快速原型。

### 第二层：三标记自检

Worker 完成编码后的**强制**验证：

| 标记          | 含义         | 方式             |
| ------------- | ------------ | ---------------- |
| `LINT_PASS`   | 代码检查通过 | 运行 lint 工具   |
| `TEST_PASS`   | 全量测试通过 | 0 failures       |
| `MANUAL_PASS` | 手动验证通过 | 逐条对照验收标准 |

### 第三层：外部 CLI 审查

中高风险任务由外部 AI 进行**独立代码审查**（ABCDE 分级）：

| 等级 | CLI 数量 | 适用场景                  |
| ---- | -------- | ------------------------- |
| A ⚫ | 3 个 CLI | 影响框架核心 / 不可逆变更 |
| B 🔴 | 2 个 CLI | 跨模块 / 用户可感知功能   |
| C 🟡 | 1 个 CLI | 单模块 / 内部逻辑变更     |
| D 🟢 | 0        | 低风险 / 测试 / 配置      |
| E    | 0        | 文档 / 注释 / 格式        |

### 第四层：PM 三阶段验收

1. **Stage 1**：规范合规（代码是否符合项目规范）
2. **Stage 2**：代码质量（逻辑正确性、边界处理）
3. **Stage 3**：文档沉淀（重要发现是否已记录）

## 6. 记忆系统

easyAI 不会因为关闭会话而遗忘。记忆系统包含：

### journal 日志

每次会话的关键发现、决策、变更都沉淀为 journal 条目（存放在 `.trellis/workspace/` 下）。新会话启动时自动检索相关条目恢复上下文。

### 会话管理

- **启动**：PM/Worker 启动时自动获取项目状态、检索最新日志、恢复上下文
- **收尾**：会话结束时自动汇总工作、写入 journal、Git 提交、输出恢复指引
- **跨会话**：支持按标签、关键词、日期搜索历史记录

## 7. 知识图谱

`.agents/graph/` 目录包含了一棵**五层 × 20 节点的知识树**，记录框架所有特性的依赖关系、文件映射和层级归属。

### 三层自维护机制

1. **编辑前**：`framework-edit-guard` Rule 强制在修改框架文件前查图谱
2. **执行后**：`worker-session-close` 自动扫描 git diff，检查图谱是否需要更新
3. **发布前**：`/publish` 工作流执行图谱一致性校验

### 渐进式披露

```text
Level 0 · 看 _index.md     →  五层域名称（一屏概览）
Level 1 · 进入某个域        →  域下所有特性节点及描述
Level 2 · 打开某个节点      →  关联关系、实现文件、变更影响
```

## 8. 约束分层：五级优先级

easyAI 的所有行为约束分五个层级，高层覆盖低层：

| 优先级    | 层级     | 类型   | 示例                                   |
| --------- | -------- | ------ | -------------------------------------- |
| 1（最高） | Rules    | 硬约束 | 反幻觉、编码规范 — 始终生效            |
| 2         | Skills   | 硬约束 | TDD 流程、验证标记 — 角色激活时生效    |
| 3         | MCP      | 硬约束 | 状态机校验、Evidence Gate — 运行时拦截 |
| 4         | PM 判断  | 软约束 | 约束集、任务指令 — 可由用户调整        |
| 5（最低） | 用户偏好 | 软约束 | 个人习惯 — 可被上层覆盖                |

**冲突解决**：硬约束（1-3）不可豁免，即使用户要求也不行。软约束（4-5）冲突时由用户裁决。
