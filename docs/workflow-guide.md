# 📋 工作流实战指南

> **预计阅读时间**：12 分钟 | **目标**：掌握 easyAI 日常操作和配置定制

---

## 1. 日常工作流全景

```text
启动 PM（/actor-pm）
    ↓
提出需求 → PM 发散讨论 → 生成设计文档
    ↓
PM 拆分任务 → 你审批
    ↓
启动 Worker（/actor-worker T001）→ 自动编码+测试
    ↓
回到 PM 会话 → PM 验收
    ↓
说「收工」→ PM 自动保存进度、Git 提交
```

---

## 2. PM 操作手册

### 启动会话

```text
/actor-pm
```

PM 会自动获取项目状态并展示：当前分支、活跃任务、上次会话摘要。

### 可用场景

| 你说什么       | PM 做什么                                               |
| -------------- | ------------------------------------------------------- |
| 描述一个新功能 | 激活需求发散 → 苏格拉底式提问 → 生成设计文档 → 拆分任务 |
| 「验收 T001」  | 三阶段审查（规范合规 → 代码质量 → 文档沉淀）            |
| 「查看任务」   | 列出所有活跃任务及状态                                  |
| 「进度归档」   | 清理已完成任务、归档文档、生成阶段总结                  |
| 「收工」       | 汇总会话成果、写入日志、Git 提交、输出恢复指引          |

### PM 不能做什么

- ❌ 直接写代码（会委派给 Worker）
- ❌ 跳过需求讨论直接创建任务
- ❌ 未经你确认就 push 代码

---

## 3. Worker 操作手册

### 启动 Worker

```text
/actor-worker T001
```

> 需要在**新的对话**中启动。Worker 和 PM 运行在不同会话中。

### Worker 会做什么

1. **读取约束集**：从 `task.md` 中加载所有约束和验收标准
2. **TDD 编码**：先写测试 → 写实现 → 重构
3. **三标记自检**：Lint ✅ + Test ✅ + Manual ✅
4. **提交验收**：状态变为 `under_review`

### 执行模式

| 模式     | 适用场景   | 说明                            |
| -------- | ---------- | ------------------------------- |
| 独立执行 | 大多数情况 | 单个 Worker 完成全部任务        |
| 团队模式 | 复杂任务   | 组长拆分子任务给多个组员 Worker |
| 轻量修复 | 简单修改   | 跳过 TDD，快速修复              |

### Worker 遇到问题怎么办？

Worker 按问题严重程度分级处理：

- **L1（轻度）**：5 分钟能解决 → 自行处理
- **L2（中度）**：不阻塞整体 → 记录后继续
- **L3（重度）**：约束歧义 / 架构不确定 → **停止工作**，创建 blocker 文件上报 PM

## 4. 会话管理：跨会话记忆的秘密

easyAI 最强大的能力之一是**跨会话不失忆**。但这需要你配合做一件事——**收工**。

### 启动会话

```text
/actor-pm
```

PM 自动恢复上下文：

- 读取最新 journal 日志（上次做了什么、遗留了什么）
- 列出活跃任务及其状态
- 检查 PM 记事本中的待办事项
- 检测语义地图是否需要刷新

### ⭐ 收工（每次结束工作时必做）

```text
收工
```

收工是 easyAI **记忆系统的写入操作**。就像保存文件一样，不做就丢失。PM 会：

| 步骤 | 做什么                | 为什么重要                                   |
| ---- | --------------------- | -------------------------------------------- |
| 1    | **汇总本次工作**      | 提取关键发现、决策、遗留事项                 |
| 2    | **写入 journal 日志** | 持久化到 `.trellis/workspace/`，下次自动恢复 |
| 3    | **Git 自动提交**      | 保存所有文件变更，可回滚                     |
| 4    | **输出恢复指引**      | 告诉 AI 下次恢复时注意什么                   |

> ⚠️ **记住**：不收工 = 下次 AI 丢失本次上下文。习惯刚开始可能会忘，但坏一次就会记住了。

### 中途暂停

直接关闭会话也可以，但**建议先说「收工」再关**。如果来不及收工就被打断了，下次启动 PM 时 AI 会尽力从 Git 状态和文件系统恢复，但效果不如正式收工。

## 5. 配置定制

框架配置文件位于 `.trellis/config/config.yaml`。以下是你可能需要调整的配置：

### 开发者信息

```yaml
developer:
  name: "你的名字" # 用于 journal 日志归属
```

### 用户交互级别

```yaml
user_level: intermediate # novice | intermediate | expert
```

AI 会根据此配置调整输出的详细度。

### 外部 CLI

```yaml
external_cli: [] # 可配置 codex / claude / gemini
```

如果安装了外部 AI CLI 工具，在这里声明后，PM 可以调用它们进行独立代码审查。

### 核心规则

```yaml
core_rules: []
# 示例: ["所有 API 必须有错误处理", "禁止使用 any 类型"]
```

PM 和 Worker 会将这些规则作为硬性约束遵守。

## 6. 框架升级

```bash
npx @geeseeker/easyai-dev update .
```

升级机制基于 **Manifest 智能合并**：

- ✅ 未修改的框架文件 → 直接替换为新版本
- ✅ 你自定义的 Skills/Rules/Workflows → 不会被触碰
- ⚠️ 你修改过的框架文件 → 生成 `.new` 冲突文件，由你决定合并方式

> 升级后需重启 IDE 以使 MCP Server 更新生效。

## 7. ⭐ 进度归档

归档是比收工更大粒度的操作。当完成一个**里程碑**（版本发布、大功能上线、项目阶段性完成）时：

```text
进度归档
```

PM 执行全套清理流程：

| 步骤 | 做什么           | 细节                                                                             |
| ---- | ---------------- | -------------------------------------------------------------------------------- |
| 1    | **设计文档打标** | `.docs/design/features/` 中已完成的文档打上 `[DONE]` 前缀，过时的打 `[OBSOLETE]` |
| 2    | **记事本去水**   | `.docs/notes/` 中已处理的待办项清除，过期草稿归档                                |
| 3    | **任务归档**     | 已完成的任务移入 `.trellis/tasks/archive/`                                       |
| 4    | **临时文件清理** | `.tmp/` 目录清空                                                                 |
| 5    | **里程碑总结**   | 生成阶段性成果报告，沉淀到 journal                                               |

### 收工 vs 归档 — 什么时候用哪个？

|            | 收工               | 进度归档                           |
| ---------- | ------------------ | ---------------------------------- |
| 频率       | **每次**结束工作   | 完成**里程碑**时                   |
| 比喻       | 下班打卡           | 季度总结                           |
| 做什么     | 写日志 + Git 提交  | 全套文档清理 + 任务归档 + 总结报告 |
| 不做会怎样 | 下次可能丢失上下文 | 废旧文档越积越多，影响效率         |

## 8. 常见问题 FAQ

### MCP Server 报「无法找到项目根目录」

v4.2.5+ MCP Server 通过 **roots 协议自动检测** IDE 当前项目，通常不会出现此问题。

如果仍然出现，可在 MCP 配置中添加环境变量作为后备：

```json
{
  "easyai-mcp-server": {
    "command": "npx",
    "args": ["-y", "@geeseeker/easyai-dev", "serve"],
    "env": {
      "EASYAI_PROJECT_ROOT": "/你的项目绝对路径"
    }
  }
}
```

修改后重启 IDE。

### 切换项目时怎么办？

**无需任何操作**。MCP Server 通过 roots 协议自动检测 IDE 当前打开的项目，切换项目时零配置。

### Worker 连续失败怎么办？

Worker 连续 3 次修复失败会自动停下，生成失败报告上报 PM。你可以在 PM 会话中查看报告，调整策略后重新分配任务。

### 如何检查框架完整性？

```bash
npx @geeseeker/easyai-dev check .
```

如果检查失败，可以重新初始化：

```bash
npx @geeseeker/easyai-dev init .
```

### PM 和 Worker 可以在同一个会话中吗？

不建议。PM 和 Worker 设计为**独立会话运行**，这确保了角色隔离和质量闭环。PM 会话中创建任务，Worker 在新会话中执行。

### 如何添加自己的规范？

在 `.trellis/spec/` 目录下创建 Markdown 文件。PM 和 Worker 在编码前会自动加载这些规范。

支持的分类目录：

- `spec/frontend/` — 前端规范
- `spec/backend/` — 后端规范
- `spec/guides/` — 通用指南
- `spec/general/` — 全局规范
