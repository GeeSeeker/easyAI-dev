# 📋 工作流实战指南

> **预计阅读时间**：12 分钟 | **目标**：掌握 easyAI 日常操作和配置定制

---

## 1. 日常工作流全景

```text
启动 PM（/actor-pm）
    ↓
提出需求 → PM 发散讨论 → 生成设计文档
    ↓
PM 拆分任务 → 你审批
    ↓
启动 Worker（/actor-worker T001）→ 自动编码+测试
    ↓
回到 PM 会话 → PM 验收
    ↓
说「收工」→ PM 自动保存进度、Git 提交
```

---

## 2. PM 操作手册

### 启动会话

```text
/actor-pm
```

PM 会自动获取项目状态并展示：当前分支、活跃任务、上次会话摘要。

### 可用场景

| 你说什么       | PM 做什么                                               |
| -------------- | ------------------------------------------------------- |
| 描述一个新功能 | 激活需求发散 → 苏格拉底式提问 → 生成设计文档 → 拆分任务 |
| 「验收 T001」  | 三阶段审查（规范合规 → 代码质量 → 文档沉淀）            |
| 「查看任务」   | 列出所有活跃任务及状态                                  |
| 「进度归档」   | 清理已完成任务、归档文档、生成阶段总结                  |
| 「收工」       | 汇总会话成果、写入日志、Git 提交、输出恢复指引          |

### PM 不能做什么

- ❌ 直接写代码（会委派给 Worker）
- ❌ 跳过需求讨论直接创建任务
- ❌ 未经你确认就 push 代码

## 3. Worker 操作手册

### 启动 Worker

```text
/actor-worker T001
```

> 需要在**新的对话**中启动。Worker 和 PM 运行在不同会话中。

### Worker 会做什么

1. **读取约束集**：从 `task.md` 中加载所有约束和验收标准
2. **TDD 编码**：先写测试 → 写实现 → 重构
3. **三标记自检**：Lint ✅ + Test ✅ + Manual ✅
4. **提交验收**：状态变为 `under_review`

### 执行模式

| 模式     | 适用场景   | 说明                            |
| -------- | ---------- | ------------------------------- |
| 独立执行 | 大多数情况 | 单个 Worker 完成全部任务        |
| 团队模式 | 复杂任务   | 组长拆分子任务给多个组员 Worker |
| 轻量修复 | 简单修改   | 跳过 TDD，快速修复              |

### Worker 遇到问题怎么办？

Worker 按问题严重程度分级处理：

- **L1（轻度）**：5 分钟能解决 → 自行处理
- **L2（中度）**：不阻塞整体 → 记录后继续
- **L3（重度）**：约束歧义 / 架构不确定 → **停止工作**，创建 blocker 文件上报 PM

## 4. 会话管理最佳实践

### 开始一天的工作

```text
/actor-pm
```

PM 自动恢复上下文：

- 读取最新 journal 日志
- 列出活跃任务
- 展示上次会话摘要

### 中途暂停

直接关闭会话即可，不需要特殊操作。下次启动 PM 时会自动恢复。

如果想要更好的恢复效果，可以说「收工」，PM 会：

- 汇总本次工作内容
- 写入 journal 日志（持久化记忆）
- Git 自动提交
- 输出恢复指引

### 日志搜索

PM 可以搜索历史日志：

```text
帮我查一下上周关于登录功能的讨论记录
```

PM 会调用 journal 搜索工具，按标签和关键词检索。

## 5. 配置定制

框架配置文件位于 `.trellis/config/config.yaml`。以下是你可能需要调整的配置：

### 开发者信息

```yaml
developer:
  name: "你的名字" # 用于 journal 日志归属
```

### 用户交互级别

```yaml
user_level: intermediate # novice | intermediate | expert
```

AI 会根据此配置调整输出的详细度。

### 外部 CLI

```yaml
external_cli: [] # 可配置 codex / claude / gemini
```

如果安装了外部 AI CLI 工具，在这里声明后，PM 可以调用它们进行独立代码审查。

### 核心规则

```yaml
core_rules: []
# 示例: ["所有 API 必须有错误处理", "禁止使用 any 类型"]
```

PM 和 Worker 会将这些规则作为硬性约束遵守。

## 6. 框架升级

```bash
npx @geeseeker/easyai-dev update .
```

升级机制基于 **Manifest 智能合并**：

- ✅ 未修改的框架文件 → 直接替换为新版本
- ✅ 你自定义的 Skills/Rules/Workflows → 不会被触碰
- ⚠️ 你修改过的框架文件 → 生成 `.new` 冲突文件，由你决定合并方式

> 升级后需重启 IDE 以使 MCP Server 更新生效。

---

## 7. 进度归档

当完成一个大阶段时，可以告诉 PM：

```text
进度归档
```

PM 会执行清理流程：

1. **文档清理**：`.docs/notes/` 中的过期草稿标记归档
2. **冷藏区转移**：已完成的设计文档移到 `.docs/archive/`
3. **临时文件清理**：`.tmp/` 目录清空
4. **任务归档**：已完成的任务移到 `archive/`
5. **阶段总结**：生成里程碑总结报告

## 8. 常见问题 FAQ

### MCP Server 报「无法找到项目根目录」

**最常见的问题**。检查 MCP 配置中是否设置了 `EASYAI_PROJECT_ROOT`：

```json
{
  "easyai-mcp-server": {
    "command": "npx",
    "args": ["-y", "@geeseeker/easyai-dev", "serve"],
    "env": {
      "EASYAI_PROJECT_ROOT": "/你的项目绝对路径"
    }
  }
}
```

修改后需重启 IDE。

### 切换项目时怎么办？

更新 MCP 配置中的 `EASYAI_PROJECT_ROOT` 为新项目路径，重启 IDE。

### Worker 连续失败怎么办？

Worker 连续 3 次修复失败会自动停下，生成失败报告上报 PM。你可以在 PM 会话中查看报告，调整策略后重新分配任务。

### 如何检查框架完整性？

```bash
npx @geeseeker/easyai-dev check .
```

如果检查失败，可以重新初始化：

```bash
npx @geeseeker/easyai-dev init .
```

### PM 和 Worker 可以在同一个会话中吗？

不建议。PM 和 Worker 设计为**独立会话运行**，这确保了角色隔离和质量闭环。PM 会话中创建任务，Worker 在新会话中执行。

### 如何添加自己的规范？

在 `.trellis/spec/` 目录下创建 Markdown 文件。PM 和 Worker 在编码前会自动加载这些规范。

支持的分类目录：

- `spec/frontend/` — 前端规范
- `spec/backend/` — 后端规范
- `spec/guides/` — 通用指南
- `spec/general/` — 全局规范
