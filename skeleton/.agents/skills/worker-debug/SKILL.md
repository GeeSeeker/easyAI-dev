# 系统性调试（Break-Loop）

## 适用场景

Worker 遇到以下情况时，**必须**激活本 Skill：

- 测试失败
- Bug 复现
- 意外行为
- 构建失败
- 性能问题
- 集成问题

## Iron Law

```
没有根因调查，不准尝试修复
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你没有完成 Phase 1，你就不能提出修复方案。

### 常见合理化借口 vs 真相

| 借口 | 真相 |
|------|------|
| "快速修一下，以后再调查" | 症状修复会留下技术债和隐藏 Bug |
| "试试改改 X 看看行不行" | 随机修复浪费时间且无法确认根因 |
| "一次加多个改动会更快" | 无法隔离有效变更，调试更困难 |
| "跳过测试，我手动验证过了" | 手动验证无记录、不可重复、不可信 |
| "我确定就是这个原因" | 确定 ≠ 证据，需要数据支撑 |

## 四阶段流程

### Phase 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠地触发？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个问题？
   - `git diff`、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统的证据收集**

   当系统有多个组件时：

   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传播
     - 检查每层的状态

   运行一次收集证据 → 分析证据定位故障组件 → 调查该组件
   ```

5. **追踪数据流**
   - 错误值从哪里起源？
   - 谁用错误值调用了这个函数？
   - 不断向上追踪直到找到源头
   - 在源头修复，不在症状处修复

### Phase 2：模式分析

1. **找到工作正常的例子** — 同一代码库中类似的正常代码
2. **与参考对比** — 如果在实现某个模式，完整阅读参考实现（不能略读）
3. **识别差异** — 正常和异常之间有什么不同？列出所有差异，无论多小
4. **理解依赖** — 这个组件需要什么？什么设置、配置、环境？

### Phase 3：假设验证

1. **形成单一假设** — 明确陈述："我认为 X 是根因，因为 Y"
2. **最小测试** — 做最小的变更来验证假设。一次一个变量。
3. **验证后再继续** — 有效？→ Phase 4。无效？→ 形成新假设。不要叠加修复。

### Phase 4：实施修复

1. **创建失败测试用例**
   - 最简单的复现
   - 自动化测试优先
   - 使用 `worker-implement` Skill 的 TDD 流程

2. **实施单一修复**
   - 解决已识别的根因
   - 一次只做一个变更
   - 不要"顺便"改进其他东西

3. **验证修复**
   - 测试通过？
   - 其他测试没被破坏？
   - 问题是否真正解决？

4. **如果修复无效 — 计数器**

   | 失败次数   | 行动                           |
   | ---------- | ------------------------------ |
   | < 3 次     | 返回 Phase 1，用新信息重新分析 |
   | **≥ 3 次** | **停下！质疑架构（见下文）**   |

### 承认不确定性

如果经过 4 阶段调查仍无法确定根因：

- **明确声明**："经过 N 轮调查，我无法确定根因"
- **记录已排除的假设**：列出所有已验证但排除的可能性
- **分类为**：`环境问题` / `数据依赖` / `时序竞争` / `未知根因`
- **上报 PM** 并附完整调查日志

## 3 次失败上报规则

**修复 3 次仍然失败时，必须停下：**

检测到以下模式说明是架构问题：

- 每次修复暴露新的共享状态 / 耦合 / 不同位置的问题
- 修复需要"大规模重构"才能实施
- 每次修复在其他地方产生新症状

**此时必须：**

1. 停止修复尝试
2. 质疑根本方案 — 这个模式本身是否合理？
3. 生成 `dev/failure-report.md`（见下文）
4. 上报 PM，等待架构决策

**这不是失败的假设 — 这是错误的架构。**

## 5 维分析

每次调试完成后（无论成功失败），产出 5 维分析：

| 维度             | 内容                                                 |
| ---------------- | ---------------------------------------------------- |
| **根因分类**     | 类型错误 / 逻辑错误 / 配置错误 / 架构缺陷 / 环境问题 |
| **为何之前失败** | 之前尝试失败的原因分析                               |
| **预防机制**     | 如何防止同类问题再次发生                             |
| **同类检查**     | 代码库中是否存在类似隐患                             |
| **知识沉淀**     | 通过 `journal_append()` 记录学到的教训               |

## failure-report.md 模板

3 次修复失败时，生成 `dev/failure-report.md`：

```markdown
# 失败报告

## 任务信息

- 任务 ID：{task_id}
- 报告时间：{timestamp}
- 修复尝试次数：{count}

## 问题描述

{问题的清晰描述}

## 尝试记录

### 尝试 1

- 假设：{假设内容}
- 修改：{具体变更}
- 结果：{失败原因}

### 尝试 2

...

### 尝试 3

...

## 架构质疑

- 当前方案是否从根本上不可行？
- 是否需要重新设计？
- 建议的替代方案：

## 需要 PM 决策

1. {决策点 1}
2. {决策点 2}
```

## Red Flags — 立即停下

如果你在想：

- "快速修一下，以后再调查"
- "试试改改 X 看看行不行"
- "一次加多个改动然后跑测试"
- "跳过测试，我手动验证"
- "大概是 X，让我修了"
- "我不完全理解但这可能行"
- "再试一次修复"（已经试了 2 次以上）

**所有这些都意味着：停下。返回 Phase 1。**
