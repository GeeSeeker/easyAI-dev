# PM 需求发散（苏格拉底式）

## 适用场景

PM 接收到用户新需求时，**必须**先激活本 Skill 完成需求澄清和设计确认，然后才能创建任务。

## HARD-GATE

```
设计未经用户确认前，不得创建任务、不得开始实现。
此规则适用于所有项目，无论看起来多"简单"。
```

## 反面模式："太简单不需要设计"

每个需求都要经过此流程。即使是 TODO 列表、单函数工具、配置变更——都要。

"简单"项目正是未经审视的假设造成最多浪费工作的地方。设计可以简短（真正简单的项目只需几句话），但**必须呈现给用户并获得批准**。

## 流程

### Step 1：探索项目上下文

- 检查当前项目状态（文件、文档、最近提交）
- 调用 `project_status()` 获取全局状态
- 了解需求涉及的技术领域

### Step 2：引导式提问

- **一次只问一个问题** — 不要用多个问题轰炸用户
- 优先使用**选择题**（更容易回答），必要时用开放题
- 聚焦理解：目的、约束、成功标准
- 保持苏格拉底式态度：引导用户思考，不替用户做产品决策

**引导方向**：

- 谁是这个功能的用户？
- 什么场景下会使用？
- 成功是什么样子的？
- 有什么限制或约束？
- 与现有功能如何关联？

### Step 3：方案探索

- 提出 **2-3 个不同方案**，附带权衡分析
- 先推荐一个方案并解释理由
- 不同方案之间对比维度：复杂度、实现时间、可扩展性、风险

### Step 4：分段呈现设计

- 一旦认为理解了要构建什么，呈现设计
- 每个部分按复杂度缩放：简单的几句话，复杂的 200-300 词
- **每部分呈现后询问用户确认**
- 覆盖：架构、组件、数据流、错误处理、测试策略

### Step 5：用户确认

- 用户确认设计后，转化为约束集格式
- 约束集必须精准到执行者可"零决策"机械执行

### Step 6：持久化设计（HARD-GATE #2）

> **Artifacts 沉淀**：这一步的本质是将 Artifacts 中的 `implementation_plan` 提炼并持久化到 `.docs/design/`，使设计成果从会话热记忆沉淀为项目持久文档。

设计确认后，**必须**先持久化为设计文档，再创建任务：

1. 将确认的设计写入 `.docs/design/{feature-name}.md`
2. 设计文档必须包含：
   - 方案选择及理由
   - 已排除方案及排除原因
   - 约束条件和边界
   - 验收标准草案
3. 确认设计文档已保存后，进入 Step 7 创建任务

> ⚠️ 没有持久化的设计文档就调用 `task_create()` = 违规。口头确认不算持久化。

### Step 7：转入任务创建

- 调用 `task_create(role: "pm", title: "...", description: "（引用设计文档路径）...", ...)` 创建任务
- 在 task description 中引用设计文档路径
- 约束集写入 task.md
- YAGNI 原则：从设计中移除不必要的功能

## 关键原则

- **一次一个问题** — 不要用问题列表轰炸用户
- **选择题优先** — 比开放题更容易回答
- **YAGNI 无情** — 从所有设计中移除不必要的功能
- **探索替代方案** — 给出方案前总是提出 2-3 个选择
- **增量验证** — 呈现设计，获得批准后再继续
- **灵活调整** — 某部分不清楚时回头澄清

## PATEOAS 导航

```markdown
### 下一步行动

1. [ ] 完成需求澄清（一次一个问题）
2. [ ] 提出 2-3 个方案及权衡
3. [ ] 分段呈现设计并获得用户确认
4. [ ] 将确认的设计写入设计文档（HARD-GATE #2）
5. [ ] 调用 `task_create(role: "pm", title: "...", ...)` 创建任务

### 状态快照

- 当前角色：PM
- 当前阶段：BRAINSTORM
- 需求澄清：{进行中/已完成}
- 方案探索：{未开始/进行中/已确定}
- 设计确认：{未开始/部分确认/全部确认}
```
