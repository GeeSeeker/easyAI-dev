# PM 需求发散（苏格拉底式）

## 适用场景

PM 接收到用户新需求时，**必须**先激活本 Skill 完成需求澄清和设计确认，然后才能创建任务。

## HARD-GATE

```
设计未经用户确认前，不得创建任务、不得开始实现。
此规则适用于所有项目，无论看起来多"简单"。
```

## 反面模式："太简单不需要设计"

每个需求都要经过此流程。即使是 TODO 列表、单函数工具、配置变更——都要。

"简单"项目正是未经审视的假设造成最多浪费工作的地方。设计可以简短（真正简单的项目只需几句话），但**必须呈现给用户并获得批准**。

## 流程

### Step 1：探索项目上下文

- 检查当前项目状态（文件、文档、最近提交）
- 调用 `project_status()` 获取全局状态
- 扫描 `.trellis/spec/` 了解现有规范覆盖情况
- 如新需求涉及已有规范覆盖的领域，在后续方案设计中主动遵循
- 了解需求涉及的技术领域

### Step 2：引导式提问

- **一次只问一个问题** — 不要用多个问题轰炸用户
- 优先使用**选择题**（更容易回答），必要时用开放题
- 聚焦理解：目的、约束、成功标准
- 保持苏格拉底式态度：引导用户思考，不替用户做产品决策

**引导方向**：

- 谁是这个功能的用户？
- 什么场景下会使用？
- 成功是什么样子的？
- 有什么限制或约束？
- 与现有功能如何关联？

### Step 3：方案探索

- 提出 **2-3 个不同方案**，附带权衡分析
- 先推荐一个方案并解释理由
- 不同方案之间对比维度：复杂度、实现时间、可扩展性、风险

### Step 3.5：自评 ABCDE 工作分级 + CLI 可行性分析

对推荐方案自评 ABCDE 工作重要性等级（参照 `common-skill-eval.md` Step 3 的定义）。

**CLI 可行性分析触发规则**：

- **A/B 级方案**：**必须**在呈现用户前调用外部 CLI 进行可行性分析。使用 `analyze` 模式（非 `review` 模式），让 CLI 基于原始需求和项目规范独立判断方案可行性，避免受 PM 已有分析的锚定偏差影响。
- **C 级及以下**：可直接呈现用户，无需 CLI 分析。

**A/B 级方案的 CLI 分析流程**：

1. 激活 `common-cli-dispatch` Skill，指定 `--context-mode analyze`
2. 提供：原始用户需求文本 + 项目当前 spec 摘要 + PM 推荐的 2-3 个方案概要
3. **屏蔽**：PM 的详细分析过程和偏好判断（让 CLI 独立思考）
4. 要求 CLI 输出：
   - 各方案的技术可行性评估
   - 潜在风险和盲点
   - 推荐方案及理由（可能与 PM 不同）
5. CLI 分析报告写入 `cli/` 目录（若有活跃 PM 任务则写入任务目录，否则写入 `.tmp/`）

> 自评结果和 CLI 分析摘要一并附在方案呈现中，供用户参考。

### Step 3.6：整合 CLI 分析结果

> 仅当 Step 3.5 触发了 CLI 可行性分析时执行。

1. 读取 CLI 分析报告，执行法官裁决（参照 `common-cli-dispatch` Step 5 裁决流程）
2. 对 CLI 提出的每条 finding 标记 `[ACCEPTED]` 或 `[REJECTED_CLI_ADVICE: 理由]`
3. 根据裁决结果调整方案呈现：
   - CLI 与 PM 推荐一致 → 强化推荐理由
   - CLI 推荐不同方案 → 将两种视角并列呈现用户，由用户决策
   - CLI 发现 PM 遗漏的风险 → 补充到方案权衡分析中
4. 在方案呈现中明确标注：「以下分析已经过外部 CLI 独立可行性评估」

### Step 4：分段呈现设计

- 一旦认为理解了要构建什么，呈现设计
- 每个部分按复杂度缩放：简单的几句话，复杂的 200-300 词
- **每部分呈现后询问用户确认**
- 覆盖：架构、组件、数据流、错误处理、测试策略

### Step 5：用户确认

- 用户确认设计后，转化为约束集格式
- 约束集必须精准到执行者可"零决策"机械执行

### Step 6：持久化设计（HARD-GATE #2）

> **Artifacts 沉淀**：这一步的本质是将 Artifacts 中的 `implementation_plan` 提炼并持久化到 `.docs/design/`，使设计成果从会话热记忆沉淀为项目持久文档。

设计确认后，**必须**先持久化为设计文档，再创建任务：

> **Spec 演进判断**：如果设计中包含可复用的通用决策（如 API 错误处理规范、
> 命名约定、技术选型标准），应提议将其提炼到 `.trellis/spec/` 成为项目规范，
> 通过 `common-spec-update` Skill 执行。这是 spec 内容持续成长的主要入口。

1. 将确认的设计写入 `.docs/design/{feature-name}.md`
2. 设计文档必须包含：
   - 方案选择及理由
   - 已排除方案及排除原因
   - 约束条件和边界
   - 验收标准草案
   - **CLI 分析摘要**（若 Step 3.5 触发了 CLI 分析）：包含 CLI 的独立评估结论和 PM 的裁决结果（`[ACCEPTED]` / `[REJECTED_CLI_ADVICE]` 标记）
3. 确认设计文档已保存后，进入 Step 7 创建任务

> ⚠️ 没有持久化的设计文档就调用 `task_create()` = 违规。口头确认不算持久化。

### Step 7：转入任务规划

- 激活 `pm-task-planning` Skill 将设计文档转化为约束集任务
- 在 task description 中引用设计文档路径
- YAGNI 原则：从设计中移除不必要的功能

## 关键原则

- **一次一个问题** — 不要用问题列表轰炸用户
- **选择题优先** — 比开放题更容易回答
- **YAGNI 无情** — 从所有设计中移除不必要的功能
- **探索替代方案** — 给出方案前总是提出 2-3 个选择
- **增量验证** — 呈现设计，获得批准后再继续
- **灵活调整** — 某部分不清楚时回头澄清

## PATEOAS 导航

```markdown
### 下一步行动

1. [ ] 完成需求澄清（一次一个问题）
2. [ ] 提出 2-3 个方案及权衡
3. [ ] ABCDE 自评 + CLI 可行性分析（A/B 级触发）
4. [ ] 分段呈现设计并获得用户确认
5. [ ] 将确认的设计写入设计文档（HARD-GATE #2）
6. [ ] 激活 `pm-task-planning` Skill 遍历约束集和任务创建流程

### 状态快照

- 当前角色：PM
- 当前阶段：BRAINSTORM
- 需求澄清：{进行中/已完成}
- 方案探索：{未开始/进行中/已确定}
- CLI 可行性分析：{未触发/进行中/已完成/不适用（C级以下）}
- 设计确认：{未开始/部分确认/全部确认}
```

## 触发测试用例

### TC-01: 新需求触发 brainstorm

- **场景**: 用户在 PM 会话中提出全新功能需求
- **输入**: 「我想给项目加一个用户认证功能」
- **期望行为**:
  1. 激活本 Skill（不跳过直接创建任务）
  2. 执行 Step 1 探索项目上下文
  3. 进入 Step 2 引导式提问（一次一个问题）
  4. 在 Step 6 将设计持久化到 `.docs/design/` 后激活 `pm-task-planning`
- **验证方法**: 检查是否产出了 `.docs/design/` 下的设计文档，且最终激活了 `pm-task-planning`（而非直接调用 `task_create()`）

### TC-02: 简单需求也走 brainstorm

- **场景**: 用户提出看似简单的配置变更
- **输入**: 「帮我改一下 config.yaml 里的超时设置」
- **期望行为**: 即使需求简单，仍执行 brainstorm 流程（设计可以简短，但必须呈现给用户确认）
- **验证方法**: 检查是否跳过了 HARD-GATE（设计未经确认就创建任务 = 违规）

### TC-03: A/B 级方案触发 CLI analyze 模式

- **场景**: PM 评估推荐方案为 A 或 B 级（高重要性）
- **输入**: 「我需要重构整个认证系统并引入 OAuth2 SSO」（预期为 A/B 级）
- **期望行为**:
  1. Step 3.5 自评为 A/B 级
  2. 激活 `common-cli-dispatch` 并使用 `--context-mode analyze`（非 review）
  3. 屏蔽 PM 的详细分析过程，仅提供原始需求和方案概要
  4. CLI 返回独立的可行性评估后，执行 Step 3.6 法官裁决
  5. 方案呈现中包含 CLI 分析摘要和裁决标记
  6. 设计文档（Step 6）中包含「CLI 分析摘要」章节
- **验证方法**: 检查是否使用了 `analyze` 模式（而非 `review`），CLI 报告写入位置正确，设计文档中包含 CLI 裁决结果

### TC-04: C 级以下方案跳过 CLI 分析

- **场景**: PM 评估推荐方案为 C 级或更低
- **输入**: 「给项目加一个 .env 文件支持」（预期为 D/E 级）
- **期望行为**: Step 3.5 自评为 C 级及以下 → 跳过 CLI 分析 → 直接呈现用户
- **验证方法**: 检查未激活 `common-cli-dispatch`，设计文档中无 CLI 分析摘要章节
