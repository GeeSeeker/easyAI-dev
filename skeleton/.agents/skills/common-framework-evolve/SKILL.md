# 框架自迭代

## 适用场景

当需要修改以下 **框架核心目录** 时，必须通过本 Skill 执行：

- `.agents/rules/` — 规则文件
- `.agents/workflows/` — 角色入口
- `.agents/skills/` — 能力模块
- `.trellis/spec/` — 项目规范（此部分也可通过 `common-spec-update` Skill 处理）
- `.trellis/config/` — 框架配置

> **注意**：`.trellis/tasks/` 和 `.trellis/workspace/` 属于运行时数据，不受本 Skill 管控。

## 执行步骤

### 1. 变更清单

列出所有将要修改的文件及变更类型：

```markdown
## 框架变更清单

| 文件                        | 变更类型 | 变更摘要      |
| --------------------------- | -------- | ------------- |
| .agents/rules/xxx.md        | 修改     | 增加 xxx 约束 |
| .agents/skills/yyy/SKILL.md | 新建     | yyy 能力模块  |
```

### 2. 影响范围分析

分析变更的影响链：

```markdown
## 影响范围报告

### 直接影响

- 变更文件数：{n} 个
- 变更类型：新建 / 修改 / 删除

### 间接影响

- 关联角色：{受影响的 PM / Worker 角色行为变更}
- 关联 Skills：{依赖受影响文件的其他 Skills}
- 约束层级：{是否影响约束优先级体系}

### 风险等级

- 低：仅新增能力，不影响现有行为
- 中：修改已有文件，可能影响角色行为
- 高：修改 Rules 或核心 Workflow，影响所有会话
```

### 3. 用户确认门

将变更清单和影响范围报告呈现给用户，**等待明确确认后才执行写入**。

**用户确认后**：

1. 逐步执行变更
2. 每步完成后验证文件完整性
3. 通过 `journal_append()` 记录变更日志

**用户拒绝**：

- 放弃全部变更，保持原文件不动
- 记录拒绝原因到 journal

### 4. 变更后验证

- 检查修改后的文件是否格式完整、语法正确
- 如涉及 MCP Server 代码，确保 `npm run build` 通过

## PATEOAS 输出

每次框架变更完成后，必须输出：

```markdown
### PATEOAS 导航

#### 下一步行动

1. [ ] 运行 `npm run build` 验证 MCP Server 构建通过（如涉及代码变更）
2. [ ] 使用 `/pm` 或 `/worker {task_id}` 继续后续任务

#### 状态快照

- 当前阶段：FRAMEWORK_EVOLVE
- 变更状态：已应用 / 已拒绝
- 变更文件数：{n} 个
- 风险等级：{低/中/高}
- 需要验证：{是/否}
```

## 硬约束

- **必须经用户确认门**：禁止绕过确认直接修改框架文件
- **必须留 journal 记录**：每次框架变更（成功或拒绝）都必须记录
- **不可批量盲改**：每个文件的变更理由必须明确，禁止"顺便改一下"
