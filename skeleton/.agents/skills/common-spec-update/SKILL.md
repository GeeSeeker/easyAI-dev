# 规范动态演进

## 适用场景

当需要新建或修改 `.trellis/spec/` 下的规范文件时，必须通过本 Skill 执行，确保变更可追溯、可回滚。

## 执行步骤

### 1. 变更前快照

- 如果是修改已有规范，先读取当前内容作为快照
- 记录当前版本号（来自 YAML frontmatter 中的 `version` 字段）

### 2. 编写变更内容

- 新建规范必须包含完整的 YAML frontmatter（title, version, category, status）
- 修改规范时递增版本号（遵循 SemVer）
- 更新 `lastUpdated` 字段为当前日期

### 3. 影响范围分析

分析此次规范变更可能影响的范围：

```markdown
## 影响范围报告

### 直接影响

- 文件：{spec文件路径}
- 变更类型：新建 / 修改 / 废弃

### 间接影响

- 关联任务：{引用此 spec 的活跃任务ID列表}
- context.jsonl 影响：{是否有冻结快照引用此 spec}

### 风险等级

- 低：仅文档更新，无代码影响
- 中：影响活跃任务的约束条件
- 高：影响多个已冻结的 context snapshot
```

### 4. 用户确认门

将影响范围报告呈现给用户，等待确认后才执行写入。

**用户确认后**：

- 调用 `spec_validate(file: "{category}/{name}")` 校验格式合规性
  - 通过 → 写入变更后的规范文件
  - 不通过 → 修正格式后重新校验，直到通过
- 通过 `journal_append()` 记录变更日志

**用户拒绝**：

- 放弃变更，保持原文件不动

## PATEOAS 输出

每次规范更新完成后（无论成功或被用户拒绝），必须输出：

```markdown
### PATEOAS 导航

#### 下一步行动

1. [ ] 使用 `spec://{category}/{name}` 读取更新后的规范验证内容
2. [ ] 如有关联任务受影响，调用 `task_get({task_id})` 检查约束集一致性
3. [ ] 如需修改其他规范，通过 `spec://{category}/{name}` 读取后按本 Skill 流程处理

#### 状态快照

- 当前阶段：SPEC_UPDATE
- 变更状态：已应用 / 已拒绝
- 影响范围：{低/中/高}
- 关联任务：{受影响的任务ID列表或"无"}
```

## 硬约束

- `.trellis/spec/` 的任何写操作必须经过本 Skill
- 禁止绕过用户确认门直接写入
- 每次变更必须留下 journal 记录

## 触发测试用例

### TC-01: 标准规范更新流程

- **场景**: PM 在 brainstorm 后判断需要新增项目规范
- **输入**: 规范文件路径 + 变更内容
- **期望行为**:
  1. 拍取变更前快照
  2. 分析影响范围
  3. 用户确认后执行变更
  4. 版本号递增
- **验证方法**: 检查是否执行了快照、影响分析、用户确认门

### TC-02: 未经用户确认的规范变更被拒绝

- **场景**: AI 尝试直接修改 spec 文件而未走确认流程
- **输入**: 直接编辑 `.trellis/spec/` 下的文件
- **期望行为**: 激活本 Skill 的用户确认门 → 阻断未确认的变更
- **验证方法**: 检查是否在用户确认前阻断了文件修改
