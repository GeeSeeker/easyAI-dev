# PM Brain Dump 同化器

## 适用场景

当用户丢来一段非结构化的文本（想法、需求碎片、灵感、技术笔记、会议纪要等），PM 激活本 Skill 快速解析并分发到框架的正确位置。

**区分 `pm-brainstorm`**：

- `pm-braindump-assimilate`：输入是**原始非结构化文本**，重点是**解析和分类路由**
- `pm-brainstorm`：输入是**结构化的需求**，重点是**方案设计和用户确认**

当 Brain Dump 中包含明确的功能需求时，分类完成后可建议用户启动 `pm-brainstorm` 进行深入设计。

## 执行步骤

### Step 1：接收输入

接受以下输入形式：

- 用户直接粘贴的文本
- 用户提供的文件路径（读取文件内容）
- 多段消息的拼合

### Step 2：AI 解析与提取

利用 AI 的自然语言理解能力，从非结构化文本中提取以下维度的信息：

| 维度      | 提取内容           | 示例                          |
| --------- | ------------------ | ----------------------------- |
| 功能需求  | 用户想要什么功能   | "需要一个自动备份功能"        |
| 技术约束  | 技术选型、性能要求 | "用 PostgreSQL，响应 < 200ms" |
| 架构决策  | 系统设计相关       | "前后端分离，API 优先"        |
| 待办事项  | 需要后续跟进的事   | "回头调研一下 Redis 方案"     |
| 风险/顾虑 | 担心的问题         | "并发量大时可能有瓶颈"        |
| 参考信息  | 外部链接、产品名   | "参考 Notion 的块编辑器"      |

### Step 3：分类路由

将提取的信息路由到框架的对应位置：

| 分类      | 路由目标                             | 动作                                         |
| --------- | ------------------------------------ | -------------------------------------------- |
| 功能需求  | `.docs/notes/pm-braindump-{date}.md` | 新建草稿文件，待后续 `pm-brainstorm` 深化    |
| 技术约束  | 提议写入 `.trellis/spec/`            | 通过 `common-spec-update` 流程（需用户确认） |
| 架构决策  | `.docs/design/`                      | 建议创建架构文档或追加到现有文档             |
| 待办事项  | `pm-记事本.md` 待办区                | 直接追加                                     |
| 风险/顾虑 | `pm-记事本.md` 待办区                | 标注 `[RISK]` 前缀追加                       |
| 参考信息  | `.docs/refs/`                        | 建议保存为参考资料                           |

> **原则**：只有「待办事项」和「风险/顾虑」自动写入记事本，其他分类均需向用户确认后再写入。

### Step 4：向用户汇报

输出分类结果表格：

```markdown
## Brain Dump 同化结果

| #   | 原文摘要 | 分类     | 路由目标               | 状态      |
| --- | -------- | -------- | ---------------------- | --------- |
| 1   | {摘要}   | 功能需求 | pm-braindump-{date}.md | ✅ 已创建 |
| 2   | {摘要}   | 待办事项 | pm-记事本.md           | ✅ 已追加 |
| 3   | {摘要}   | 技术约束 | spec/{category}/       | ⏳ 待确认 |

### 建议后续动作

1. [ ] 基于功能需求启动 `pm-brainstorm` 进行方案设计
2. [ ] 确认技术约束是否写入 spec
3. [ ] 检查参考资料是否保存到 refs
```

## 触发测试用例

### TC-01: 混合内容 Brain Dump

- **场景**: 用户粘贴一段包含需求、技术选型、待办的混合文本
- **输入**: "我想做个博客系统，用 Next.js + Prisma，回头研究下 SEO 方案，参考 Medium 的阅读体验"
- **期望行为**:
  1. 提取出 4 条信息：功能需求（博客系统）、技术约束（Next.js + Prisma）、待办（SEO 方案）、参考（Medium）
  2. 功能需求写入 `pm-braindump-{date}.md`
  3. 待办追加到 `pm-记事本.md`
  4. 技术约束和参考标记为待确认
- **验证方法**: 检查分类表格完整、草稿文件已创建、记事本已追加
