# 组长编排工作流（团队执行模式）

## 适用场景

Worker 以组长角色执行任务时，必须激活本 Skill。本 Skill 包含两个阶段：

- **Phase A: PLAN** — 分析约束集、拆分子任务、引导用户开启组员会话
- **Phase B: INTEGRATE** — 审核组员产出、集成验证、汇总报告

## 前置条件

- Worker 已完成启动流程（身份声明、Rules 加载、任务读取）
- Worker 已决策升级为团队执行模式，且已获得用户确认

---

## Phase A: 计划阶段（PLAN）

> 在 `worker-implement` 之前执行

### Step 1：任务分析

1. 读取完整 task.md 约束集
2. 识别可并行的工作单元（按模块边界、前后端分离等维度）
3. 评估工作分配：
   - 哪些部分组长自己做
   - 哪些分给组员
   - 哪些需要外部 CLI 长线协同
4. 选择工作模式（审核者 / 主力+委派 / 协调者）

**三种工作模式判断：**

| 模式          | 信号                  | 组长做什么                           | 组员做什么                      |
| ------------- | --------------------- | ------------------------------------ | ------------------------------- |
| **审核者**    | 大型可并行任务        | 拆分子任务、审核组员产出、整合交付   | 各自并行实现子任务              |
| **主力+委派** | 中等复杂度任务        | 自己做主要开发，把阻塞性工作委派出去 | 跑外部 CLI 审查、实现独立小模块 |
| **协调者**    | 需要外部 CLI 长线协作 | 管理进度、裁决/验收外部 CLI 产出     | 专门维护外部 CLI 长线任务会话   |

### Step 2：创建子任务

1. 调用 `subtask_create()` 为每个工作单元创建子任务
2. 每个子任务包含：
   - 范围描述
   - 约束条件（从父任务约束集中提取相关项）
   - 验收标准
   - 依赖关系（如有）
3. 子任务类型：编码实现 / 外部 CLI 长线协同 / 独立模块
4. 调用 `subtask_dependency_graph()` 验证无循环依赖

### Step 3：并发检查 + 引导用户

1. 读取 `config.yaml` 中 `team.roster[type=primary].concurrency`
1. 计算所需并发 = 1 (PM 会话) + 1 (组长会话) + N (组员会话数)
1. 若 所需并发 > 可用并发 → 提醒用户可分批启动
1. 输出用户操作指引：

```markdown
### 操作指引

**需要的会话数**：{total} 个

- PM 会话：1 个（已有）
- 组长会话：1 个（当前）
- 组员会话：{N} 个

**启动命令**（按顺序）：

1. 新开会话 → `/actor-worker {task_id} 组员A`
2. 新开会话 → `/actor-worker {task_id} 组员B`
   ...

**执行顺序**：{并行/有依赖则说明先后}

> ⚠️ 并发提醒：当前 IDE 并发上限为 {concurrency}，
> 所需 {total} 个会话{超限提示/未超限}。
> {如超限：建议分批启动，先完成第一批再启动第二批。}

**重要**：组员完成后请回到本会话通知我。
```

### Step 4：组长自己的实现

- 如果组长也负责部分实现 → 进入 `worker-implement` Skill
- 如果组长只负责协调（审核者模式）→ 跳过 implement，等待组员
- 跳过时在执行记录中标注：`[LEAD_SKIP_IMPLEMENT] 审核者模式，等待组员完成`

---

## Phase B: 审核与整合阶段（INTEGRATE）

> 用户通知"组员已完成"后执行

### Step 5：读取组员产出

1. 通过 `subtask_dependency_graph({ task_id })` 获取所有子任务
2. 读取每个子任务的状态和产出
3. 检查所有子任务是否已完成
4. 如有未完成 → 告知用户：「以下子任务尚未完成：{列表}」

### Step 6：技术审核

逐个审核每个组员的产出：

1. **代码质量**：命名规范、代码风格、单一职责
2. **实现正确性**：功能是否按约束集要求实现
3. **接口一致性**：模块间接口是否匹配
4. **代码主权合规**：如有外部 CLI 产出，是否经过裁决/验证（Review 模式须逐条裁决，Execute 模式须 lint/test 验证）

审核结论分三种：

- ✅ **通过** — 代码合格
- ⚠️ **需修改** — 创建修改意见写入子任务文件，要求组员修改
- ❌ **严重问题** — 上报 PM

### Step 7：集成验证

1. 运行全量测试，确保组员产出互不冲突

```bash
npm test          # 或项目实际的测试命令
npm run lint      # 检查整体代码质量
```

2. 解决集成冲突或上报 PM
3. 记录集成验证结果

### Step 8：汇总报告

生成 `dev/report.md`：

```markdown
# 团队任务报告

## 任务信息

- 任务 ID：{task_id}
- 完成时间：{timestamp}
- 执行模式：团队执行模式
- 角色：组长

## 子任务完成情况

| 子任务 ID | 描述 | 执行者 | 状态 | 审核结论 |
| --------- | ---- | ------ | ---- | -------- |
| S001      | ...  | 组员A  | ✅   | 通过     |

## 变更摘要

| 文件    | 变更类型       | 说明       | 来源       |
| ------- | -------------- | ---------- | ---------- |
| {file1} | 新建/修改/删除 | {变更说明} | 组长/组员A |

## 集成验证

- 全量测试：{结果}
- 接口一致性：{结果}
- 代码主权合规：{结果}

## 约束集执行情况

| 约束 | 状态 | 说明 |
| ---- | ---- | ---- |
| C1   | ✅   | ...  |

## 验证标记引用

详见 `dev/verification.md`
```

完成后进入 `worker-check` Skill 做最终自检。

## 外部 CLI 使用策略参考

> 组长在 PLAN 阶段分配子任务时，需参考以下策略决定 CLI 使用方式。

### 工作重要性分级（ABCDE）

"工作"不等于"任务" — 它是任何角色产出的一个离散产出物。

| 等级  | 定义                      | CLI 审核要求       | 示例                                     |
| ----- | ------------------------- | ------------------ | ---------------------------------------- |
| **A** | 影响框架核心 / 不可逆变更 | 3 个 CLI 审核      | 修改框架核心文件、安全敏感逻辑、数据迁移 |
| **B** | 跨模块 / 用户可感知的功能 | 2 个 CLI 审核      | 新增用户可见功能、API 接口变更           |
| **C** | 单模块 / 内部逻辑变更     | 1 个 CLI 审核      | 单模块功能实现、内部重构                 |
| **D** | 低风险 / 测试 / 配置      | 可选（调用者自判） | 纯测试新增、配置调整、日志优化           |
| **E** | 文档 / 注释 / 格式        | 不需要             | 文档更新、注释修改、代码格式化           |

**分级规则：**

- 由执行工作的角色自行判断（PM 判自己的工作、Worker 判自己的工作）
- A/B 级必须在日志中记录分级理由
- 多 CLI 审核时，选择**不同的 CLI**以确保视角多样性

### CLI 选型规则（动态匹配）

框架不写死"某 CLI 做某事"的映射。选型流程：

```text
1. 读取 config.yaml → team.roster 中 type=external_cli 的条目
2. 取每个 CLI 的 strengths 列表
3. 将当前工作的特点与 strengths 做语义匹配
4. 选择匹配度最高的 CLI
5. 多 CLI 审核时，优先选择 strengths 互补的不同 CLI
```

### 三种调用粒度

| 粒度         | 描述                         | 典型场景                                   |
| ------------ | ---------------------------- | ------------------------------------------ |
| **单次调用** | 发一次指令，拿一次结果       | PM 让 CLI 审核方案、Worker 让 CLI 快速审查 |
| **多轮迭代** | 同一个 CLI 会话内反复交互    | 审核→发现问题→优化→再审核                  |
| **长线协同** | 开独立会话持续推进大规模工作 | 海量数据整理、大规模重构、批量测试修复     |

长线协同的承载者：**组员**（组长委派）或**独立执行者**（任务本身就是 CLI 长线工作）。

### 代码主权约束

**无论任何调用粒度**，外部 CLI 的产出都必须经过裁决或验证：

1. **Review 模式**：执行者须逐条裁决 CLI 建议（[ACCEPTED] / [REJECTED_CLI_ADVICE] / [PARTIALLY_ACCEPTED]），仅采纳正确且符合项目上下文的内容
2. **Execute 模式**：执行者须对 CLI 写入的代码执行验证（lint/test/manual），验证失败须回滚
3. 组员使用 CLI 产出时裁决/验证后合入，组长在整合阶段**再次审核** CLI 相关子任务的产出

### 审核分层（三层不重叠）

| 审核层 | 审核者     | 审核内容                           |
| ------ | ---------- | ---------------------------------- |
| 第一层 | 执行者自身 | 代码主权合规（裁决/验证 CLI 产出） |
| 第二层 | 组长       | 技术审核（代码质量、接口一致性）   |
| 第三层 | PM         | 产品验收（需求满足、验收标准达成） |

---

## 嵌套限制

**不允许嵌套** — 组员不能再分配子任务给其他组员，保持单层结构（组长 → 组员）。如果子任务复杂到需要进一步拆分，应上报组长重新规划，或上报 PM 拆为独立任务。

## 触发测试用例

### TC-01: 标准团队执行模式全流程

- **场景**: Worker 升级为组长，拆分子任务，组员完成后整合
- **输入**: 包含可并行功能模块的约束集
- **期望行为**:
  1. PLAN: 分析 → 创建子任务 → 输出操作指引（含并发提醒）
  2. 等待组员
  3. INTEGRATE: 读取产出 → 技术审核 → 集成验证 → 汇总报告
- **验证方法**: 检查子任务创建、操作指引格式、汇总报告完整性

### TC-02: 并发超限时正确提醒

- **场景**: 需要 6 个会话但 config.yaml concurrency 为 5
- **输入**: 5 个子任务 + PM + 组长 = 7 > 5
- **期望行为**: 操作指引中包含并发超限提醒 + 分批建议
- **验证方法**: 检查操作指引是否含超限提醒文本

### TC-03: 组员产出审核发现问题

- **场景**: 组员提交的代码不符合规范
- **输入**: 子任务产出中存在代码质量问题
- **期望行为**: Step 6 标记为"需修改"，创建修改意见写入子任务文件
- **验证方法**: 检查子任务文件中是否包含修改意见
