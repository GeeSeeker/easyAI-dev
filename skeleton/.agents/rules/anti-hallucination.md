# 反幻觉约束

本规则在所有会话中始终生效，防止 AI 产生幻觉内容。

## 硬性约束

1. **第三方库必须先查文档**：调用任何非语言内置的第三方库 API 前，必须通过 Context7 或联网检索确认当前版本的正确用法。**严禁凭底层记忆编码**。
2. **技术决策需有证据**：选择技术方案时，必须附带检索来源或实际测试结果作为支撑。
3. **禁止模糊措辞**：汇报状态或进度时，禁止使用 "should"、"probably"、"might work"、"应该可以" 等不确定表述。要么确认有效，要么明确说明未验证。

## RPI 阶段隔离（来源：GuDa-spec）

4. **规划阶段禁止写代码**：Research / Plan 阶段只产出约束集和决策，不得生成或修改生产代码。
5. **实现阶段禁止做架构决策**：Implementation 阶段严格按约束集执行，遇架构歧义必须上报 PM，不得自行裁决。
6. **反面模式自检**：输出方案时自检是否包含「多方案未选择」「推迟决策」「信息堆砌无结论」等反面模式。可调用 `plan_validate()` 自动检测。

## 代码主权（来源：GuDa-spec + easyAI 迭代）

7. **主控 AI 对 CLI 产出负验证责任**：外部 CLI（Codex / Claude Code / Gemini CLI）可直接写入项目文件，但主控 AI（Antigravity）必须对写入结果执行验证（lint / test / manual check）。验证通过方可认定为有效产出；验证失败须回滚或修正。CLI 的审查建议须经法官裁决（[ACCEPTED] / [REJECTED_CLI_ADVICE]）后方可采纳。

## 检索优先级

当需要查询技术信息时，按以下优先级使用工具：

1. **Context7**（首选）：官方文档检索
2. **联网搜索**（兜底）：通用搜索查漏补缺
3. **GitHub**（补充）：搜寻开源社区真实代码实践或排查 Issue

## 自检清单

在提交任何技术产出前，自问：

- [ ] 所有第三方 API 调用是否已查证文档？
- [ ] 是否存在"凭印象"编写的代码？
- [ ] 是否有未经验证的技术假设？
- [ ] 方案中是否存在未决定事项或推迟决策？
- [ ] 外部模型产出是否经过验证（lint/test）并通过法官裁决？
