# Skill 行为回归测试

> 通过文档 TDD 确保 Skill 行为可验证、可回归。

## 核心原则

**文档 TDD**：修改 Skill 前，先更新触发测试用例；修改后，逐一验证测试用例。

## 触发测试用例模板

每个 Skill 的 `SKILL.md` 末尾应包含以下章节：

```markdown
## 触发测试用例

### TC-{编号}: {测试场景名}

- **场景**: {描述触发此 Skill 的上下文}
- **输入**: {用户的具体操作或消息}
- **期望行为**: {Skill 应执行的具体步骤}
- **验证方法**: {如何确认行为正确}
```

## 示例：pm-session-start

```markdown
## 触发测试用例

### TC-01: 标准 PM 会话启动

- **场景**: 用户在新会话中首次调用 `/pm`
- **输入**: `/pm`
- **期望行为**:
  1. 调用 `project_status()` 获取状态
  2. 读取 `trellis://status` 和 `trellis://journal/latest`
  3. 扫描 `spec/` 目录结构
  4. 输出结构化状态快照
- **验证方法**: 检查输出是否包含分支、活跃任务、上次会话摘要、PATEOAS 导航

### TC-02: MCP 不可用时的降级

- **场景**: MCP Server 未启动或连接断开
- **输入**: `/pm`
- **期望行为**: 降级为手动 git 命令 + 文件系统读取
- **验证方法**: 检查是否输出了降级提示且状态恢复仍然完成
```

## 示例：common-session-close

```markdown
## 触发测试用例

### TC-01: 正常收工流程

- **场景**: PM 完成工作后
- **输入**: 用户说「收工」
- **期望行为**:
  1. 汇总会话工作（Step 1）
  2. 写入 journal（Step 2）
  3. 执行知识沉淀检查（Step 4）
  4. Git 提交（Step 6）
  5. 输出恢复指引（Step 8）
- **验证方法**: 检查 journal 是否写入、Git 是否提交、恢复指引是否包含硬约束检查项

### TC-02: 无变更时的收工

- **场景**: 会话中只进行了查询，未修改任何文件
- **输入**: 用户说「收工」
- **期望行为**: 跳过 Git 提交，仍然写入 journal 和输出恢复指引
- **验证方法**: 检查提交步骤显示"工作区已干净"
```

## 工作流程

### 新建 Skill 时

1. 在 SKILL.md 中先写 `## 触发测试用例`（至少 2 个：正常路径 + 异常路径）
2. 再编写 Skill 的执行步骤
3. 完成后逐一验证测试用例

### 修改 Skill 时

1. 先检查现有触发测试用例是否覆盖变更点
2. 不覆盖 → 新增测试用例
3. 执行修改
4. 验证所有测试用例（包括未修改的，确保无回归）

## 验收检查

`pm-task-review` 验收涉及 Skill 变更的任务时：

1. 检查 SKILL.md 是否包含 `## 触发测试用例` 章节
2. 检查测试用例是否覆盖了本次变更的行为
3. 检查测试用例的验证方法是否可执行
