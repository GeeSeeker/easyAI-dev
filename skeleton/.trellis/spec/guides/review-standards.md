# 审查标准规范

> **版本**：1.0.0
> **适用场景**：PM 验收任务时参照本规范进行风险分级审查

---

## 一、审核类型分类（F29）

根据任务性质确定审核重点：

| 审核类型     | 适用场景                        | Stage 2 重点检查项                                                       |
| ------------ | ------------------------------- | ------------------------------------------------------------------------ |
| **架构审核** | 新模块、接口设计、依赖变更      | 接口边界是否清晰、依赖方向是否正确、模块职责是否单一、与现有架构是否一致 |
| **代码审核** | 功能实现、Bug 修复、重构        | 代码风格一致性、边界情况处理、错误信息质量、可维护性、测试覆盖           |
| **文档审核** | Spec 更新、知识库维护、指南编写 | 与源码是否一致、术语是否准确、示例是否可运行、结构是否清晰               |
| **混合审核** | 跨层改动（代码 + 规范 + 文档）  | 按各部分分别适用对应类型的检查项，**风险等级取各部分最高值**             |

PM 在 Stage 1 开始前**必须**确定审核类型，写入验收报告头部。

## 二、风险分级（F17）

根据任务变更范围和影响自动确定风险等级：

| 风险等级    | 触发条件                                           | 审查深度                     | **强制动作**                                |
| ----------- | -------------------------------------------------- | ---------------------------- | ------------------------------------------- |
| 🟢 **低**   | 命中低风险白名单（见下方）                         | Stage 1 + Stage 2 快速检查   | 无（快速检查 = 只查 Critical 级，跳过风格） |
| 🟡 **中**   | 非低风险、非高风险的一般任务                       | Stage 1 + Stage 2 完整审查   | 无                                          |
| 🔴 **高**   | 修改框架核心文件、安全相关、跨模块改动             | Stage 1 + Stage 2 完整审查   | PM **必须**独立验证关键路径                 |
| ⚫ **极高** | 修改 `.agents/rules/`、MCP Server 核心逻辑、状态机 | Stage 1 + Stage 2 + 外部审查 | PM **必须**独立验证 + **必须**外部 CLI 审查 |

### 低风险白名单（F41）

以下类型的任务默认为低风险，Stage 2 可简化为快速检查：

- 纯文档修改（`.md` 文件，不含 Spec 规范文件）
- 注释或日志文本修改
- 配置文件中非关键参数调整（如 UI 偏好、提示文本）
- 测试文件新增（不修改生产代码）
- `.gitignore` / `.editorconfig` 等开发配置

> **注意**：Spec 规范文件（`.trellis/spec/`）不属于低风险，即使是 `.md` 格式。

### 高风险触发条件

以下任一条件成立即触发高风险：

- `file_scope` 包含 `.agents/` 下的 Skill 或 Workflow 文件
- `file_scope` 包含 `packages/mcp-server/src/` 下的核心文件
- `file_scope` 包含 `.trellis/spec/` 下的规范文件
- 任务涉及 3 个以上模块的跨模块修改
- 任务涉及认证、权限、数据安全相关逻辑

## 三、报告格式标准化（F18）

审查发现按严重程度分三级：

| 等级         | 标记     | 定义                                            | 处理方式               |
| ------------ | -------- | ----------------------------------------------- | ---------------------- |
| **Critical** | 🔴 `[C]` | 功能错误、安全漏洞、**Spec 违规**、数据丢失风险 | 必须修复才能通过       |
| **Warning**  | 🟡 `[W]` | 代码质量问题、缺少边界处理、测试不足            | 建议修复，可附条件通过 |
| **Info**     | 🟢 `[I]` | 代码风格、命名改进、可选优化                    | 记录即可，不阻断       |

> ⚠️ **Spec 违规一律为 Critical**。`pm-task-review` Stage 1 中「按严重程度判定」不适用于 Spec 违规类问题。

### 判定规则

- **存在任何 Critical** → 打回
- **仅有 Warning** → PM 判断：附条件通过（记录后续改进项）或打回
- **仅有 Info** → 通过

## 四、循环修复机制（F19）

防止「打回 → 修复 → 再打回」无限循环：

### 计数规则

- 每个 Stage 独立计数，进入新 Stage 时重置
- **存储方式**：每次打回时，PM 在 task.md 的 `## 执行记录` 部分追加：
  ```text
  - [ISO时间] 🔴 Stage {N} 打回第 {M} 次：{原因摘要}
  ```
- PM 在每轮审查开始时，计算执行记录中当前 Stage 的打回条目数

> ℹ️ 当前 MCP `task_transition(rejected)` 不会自动递增计数，PM 必须手动通过 `task_append_log()` 记录。后续迭代可考虑在 MCP 层自动化。

### 阈值与处置

| 打回次数  | 处置                                                                                                                 |
| --------- | -------------------------------------------------------------------------------------------------------------------- |
| 第 1-2 次 | 正常打回 + 详细修复建议                                                                                              |
| 第 3 次   | **强制升级**：任务状态保持 `rejected`，PM 与用户沟通决定：a) PM 直接修复 b) 重新拆分任务 c) 降低验收标准 d) 取消任务 |

> 第 3 次打回时，PM **必须**在验收报告中附加「循环分析」：为什么反复失败？是约束集不够清晰还是执行者能力不足？

## 五、硬阻断检查点补充（F28）

在 `pm-task-planning` 的 HARD-GATE 基础上，补充以下硬阻断检查点：

### 规划阶段硬阻断（pm-task-planning）

| #        | 检查点               | 触发时机  | 阻断条件                                                          |
| -------- | -------------------- | --------- | ----------------------------------------------------------------- |
| HG-1     | 设计文档持久化       | Step 1 前 | `.docs/design/` 无对应文件                                        |

| #    | 检查点         | 触发时机   | 阻断条件                                    |

## 六、Ralph Loop 系统级质控（F59）

完整的质量控制循环：

```text
Worker 提交 → PM Stage 1 (Spec) → PM Stage 2 (质量) → PM Stage 3 (沉淀)
     ↑                ↓ FAIL              ↓ FAIL
     └── rejected ←──┘                   └── rejected ←──┘
                                                ↓ 第3次
                                          强制升级（PM 介入）
```

每个审查节点的输入输出：

| 节点    | 输入                             | 输出                       | 失败处理            |
| ------- | -------------------------------- | -------------------------- | ------------------- |
| Stage 1 | task.md + verification.md + dev/ | Spec 合规报告              | rejected + 修复建议 |
| Stage 2 | 源代码 + Spec 报告               | 质量审查报告（C/W/I 分级） | rejected + 改进建议 |
| Stage 3 | Worker Artifacts                 | 沉淀到 .docs/              | 无失败（可选步骤）  |
