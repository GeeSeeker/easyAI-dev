# ADR 模板

> **版本**：1.0.0
> **存储路径**：`.docs/design/decisions/ADR-{序号}-{标题}.md`
> **适用场景**：记录影响架构或技术方向的决策

---

## 模板

```markdown
# ADR-{序号}: {决策标题}

- **日期**：{YYYY-MM-DD}
- **状态**：{Proposed / Accepted / Deprecated / Superseded by ADR-{N}}
- **关联任务**：{T001 / 无}

## 背景

{为什么需要做这个决策？当前的问题或需求是什么？}

## 决策

{最终选择了什么方案？用一句话概括。}

## 备选方案

| 方案           | 优势   | 劣势   | 结论              |
| -------------- | ------ | ------ | ----------------- |
| 方案 A: {名称} | {优势} | {劣势} | ✅ 采纳 / ❌ 放弃 |
| 方案 B: {名称} | {优势} | {劣势} | ✅ 采纳 / ❌ 放弃 |

## 影响

- **正面影响**：{这个决策带来什么好处？}
- **负面影响**：{有什么代价或限制？}
- **受影响的文件/模块**：{列出受影响的范围}

## 后续行动

- [ ] {如有后续工作，列出}
```

---

## 编号规则

- 序号从 001 开始，递增分配
- 查看 `.docs/design/decisions/` 目录取最大序号 + 1
- 如目录为空或不存在，从 001 开始
- **序号不连续时不复用**（如删除了 ADR-003，下一个仍为 004，避免引用混乱）
- 文件名格式：`ADR-{序号}-{标题短语}.md`（标题用连字符分隔）

> 完整操作规程见 `spec://general/knowledge-categories` § ADR 操作规程

## 状态流转

```text
       ┌─────────── Accepted（知识沉淀创建，用户已确认）
       │
Proposed → Accepted → (可选) Deprecated / Superseded
  │
  └──── （设计讨论阶段提案，尚未确认）
```

- **Proposed**：设计讨论阶段的初始提案，等待确认
- **Accepted**：用户确认采纳（**知识沉淀触发时直接使用此状态**，因为用户确认沉淀即为采纳）
- **Deprecated**：不再适用（需说明原因）
- **Superseded**：被新 ADR 替代（需引用新 ADR 编号）

## 触发条件

会话中做出符合以下任一条件的决策时，应创建 ADR：

1. 选择了技术栈或框架
2. 确定了架构模式（如模块划分、数据流方向）
3. 制定了安全策略或认证方案
4. 做出了性能 vs 可维护性等权衡取舍
5. 更改了已有的设计决策
